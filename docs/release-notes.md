# Release note labels

Release notes are generated by `auto` from merged PR labels. Each PR that should
appear in release notes needs two release-related decisions before merge:

1. Exactly one version label.
2. Exactly one primary release-note category label.

Use `category: other` for changes that should appear in release notes but do not
fit another category.

## Version labels

Choose exactly one:

| Label | Use when |
| --- | --- |
| `version: major` | The change is breaking for consumers or downstream app integration. |
| `version: minor` | The change adds new behavior, rule coverage, or capabilities. |
| `version: patch` | The change fixes existing behavior or makes a non-breaking maintenance update. |

## Release-note category labels

Choose exactly one category:

| Label | Use when |
| --- | --- |
| `category: rules` | Adding, updating, or regenerating autoconsent rules. |
| `category: bug` | Fixing broken behavior, including rule fixes and race-condition fixes. |
| `category: enhancement` | Adding or improving non-rule functionality. |
| `category: performance` | Improving runtime performance or measurement overhead. |
| `category: dependencies` | Updating package dependencies. |
| `category: ci` | Changing CI, release automation, or scripts used by automation. |
| `category: ai` | Changing agent instructions, workflows, or skills. |
| `category: documentation` | Documentation-only changes. |
| `category: tests` | Changing test harnesses, fixtures, or test infrastructure. Do not use this for ordinary tests added with feature or rule work. |
| `category: internal` | Internal-only maintenance that does not fit the categories above. |
| `category: other` | Changes that should appear in release notes but do not fit another category. |

## Common examples

| PR type | Labels |
| --- | --- |
| New CMP rule or new site-specific rule | `version: minor`, `category: rules` |
| Existing rule fix | `version: patch`, `category: bug` |
| Generated rule update | `version: minor` or `version: patch`, `category: rules` |
| Core API or integration breaking change | `version: major`, `category: enhancement` |
| CI workflow or release script update | `version: patch`, `category: ci` |
| Dependency update | `version: patch`, `category: dependencies` |
| Agent skill or instruction update | `version: patch`, `category: ai` |
| Documentation-only update | `version: patch`, `category: documentation` |
| Test runner, fixture, or harness update | `version: patch`, `category: tests` |
| Miscellaneous repo maintenance | `version: patch`, `category: other` |

If a PR spans multiple categories, choose the category that best describes the
user-visible release note. Mention secondary details in the PR description.

## Syncing GitHub labels

The label definitions live in `package.json` under the `auto.labels` config. If
labels are missing or descriptions/colors drift, sync them with:

```bash
npx auto create-labels
```

`auto create-labels` creates or updates labels from the config, but it does not
remove labels that are no longer listed. After syncing, remove or rename the old
unprefixed required labels (`major`, `minor`, `patch`, `rules`, `bug`,
`enhancement`, `performance`, `dependencies`, `ci`, `ai`, `documentation`,
`tests`, `internal`, and `other`) in the GitHub labels UI if they are no longer
needed.
